# AI 编程方法论

## 前提：清晰表达力

AI 编程的第一步不是学提示词工程，而是**清晰表达**。能准确描述问题、目标和约束，是利用任何 AI 工具的基础。表达夸张、缺乏严谨描述的用户，即使最强大的 AI 也无法给出有效帮助。沟通能力（表达能力 + 理解力）是所有后续建议的前提。

## 核心范式

### VibeCoding（氛围编程）

自然语言驱动的感性开发模式，核心是"说-写-改"快速迭代循环。

- **适用场景**：需求探索、快速原型、Bug 修复
- **优势**：5 分钟内生成完整 API 端点，开发节奏比传统快 5 倍
- **风险**：需求模糊时 AI 易跑偏（如"智能推荐"被误读为 ML 算法）

### Spec-Driven Development（规范驱动开发）

先写详细规格文档，再让 AI 生成代码的理性模式。

- **适用场景**：核心功能、架构设计、质量敏感模块
- **优势**：一次生成路由+服务+模型+测试四个文件，质量可控
- **代价**：前期投入大，需要写详细的接口定义/请求参数/业务逻辑

### CDD（上下文驱动开发，Conductor-Driven Development）

AI 先读完整项目上下文，自动生成规格与计划，人审核后 AI 按任务列表编码。

- **适用场景**：存量代码库迭代、架构复杂、需持续同步规范
- **核心资产**：活上下文（Spec + 代码 + 规范，持续演化）
- **优势**：自动逆向生成规格，不需要人工写 Spec
- **代价**：一次性读全仓 Token 消耗高，但可复用

### 三模式选择决策

| 阶段 | 推荐模式 | 理由 |
|------|---------|------|
| 0→1 原型 / 需求模糊 | VibeCoding | 最快出活 |
| 新功能上线长期维护 | SDD | 规格即合同，CI 自动比对 |
| 祖传代码库 / 规范缺失 | CDD | AI 反向生成规格 |

> 三种模式可混用：MVP 阶段 Vibe → 定型后切 SDD → 存量迭代用 CDD

### 混合开发四阶段策略

| 阶段 | 模式 | 目标 |
|------|------|------|
| 需求探索期 | VibeCoding | 快速验证 MVP |
| 架构设计期 | Spec-Driven | 写架构文档 + API 规格 + 数据模型 |
| 功能开发期 | 混合 | 核心 Spec + 辅助 Vibe |
| 测试优化期 | VibeCoding | 快速修 Bug |

## AI 编码语言选型

| 语言 | 级别 | 准确率 | 建议 |
|------|------|--------|------|
| Python / JS/TS / Java / SQL | S | 88-92% | 闭眼用 |
| C# / Go | A | 85-87% | 主力可用 |
| C/C++ / Rust | B | 65-70% | 骨架可用，需二次校验 |

## AI 编码架构选型

- **单体模块化 (Modulith)** → AI 只需理解一个 repo，一次生成成功率高
- 微服务 → 一次改 3-4 个 repo，AI 经常忘同步 DTO 字段
- **原则**：MVP/内部工具优先单体，真正流量 10 倍扩容再拆微服务

## AI 编码需求规范模板

结构化需求文档是 Spec-Driven 开发的核心资产，模板包含：

1. **元信息**：优先级、工作量、状态
2. **背景与目标**：问题陈述、成功指标（SMART）、非目标
3. **功能需求**：用户故事、功能清单、业务规则、边界情况
4. **技术规范**：技术栈约束、AI 编码特殊要求、接口定义、数据模型、状态机
5. **质量保障**：验收标准、测试策略、测试数据
6. **上下文与依赖**：前置依赖、相关文档、风险管理

## AIGC 编码提效体系

### 2026 年目标

- 后端研发任务量提升 10%，前端提升 20%
- 业务模块 100% AI 出码挑战
- 原始需求 → 设计 → 代码开发 → 单元测试 → 构建全流程

1. **公司 LightCode API**：准确性要求低、重复性高、Token 消耗高
2. **智谱 Pro 包年**：中型 Repo 日常开发
3. **MiniMax Max 极速版**：高级开发场景

## 三个核心原则

1. **明确分工**：不同工具用于不同场景
2. **持续反馈**：AI 生成后立即验证、纠偏
3. **知识沉淀**：经验复用、案例积累

## 踩坑记录

- AI 理解偏差：模糊需求导致方向性错误
- 依赖管理混乱：AI 生成的依赖冲突
- 测试覆盖不足：自动生成仅 60%，需人工补充
- JSON 解析不稳定：需多层级容错机制（78% → 98%）

## VibeCoding 实战教训（caoz）

### 记忆管理是第一痛点

AI 不记得过去改动会导致"鬼打墙"——修新 bug 时老问题重现，改回来新问题又回来。解决方案：
- 维护分层记忆文件（主索引 + 模块文档），不用一个大文件存所有东西
- 每次版本迭代后让 AI 更新记忆文件，去重冗余
- 这样 AI 后续处理时能"不伤害过往成果，不重现过往错误"

### AI 维护代码也需要可维护性

AI 编写的代码同样会膨胀：废弃部分堆积、复用性不足导致重复编写。后果：
- 冗余代码浪费 token
- 复用性不够导致业务逻辑不统一
- 废弃逻辑可能误导 AI 理解

解决：每次版本更新后让 AI 在"记忆最完整时"清理冗余、删除废弃代码、归并相似结构。

### 自动测试工作量不降反增

AI 编程后测试占迭代工作量的 **70%**。原因：你不知道 AI 改了什么，指令约束不够时它会乱改。自动测试只能覆盖功能正确性，体验优化（难度、流畅度、数值平衡）仍需人工。

### 早占坑策略

App Store/Google Play 因 AI 生成产品暴增而审核趋严。建议：尽早提交产品占位，后续慢慢迭代。

## 本地模型的关注方向

大模型虽好，但 API 使用成本持续上升。本地模型在成本效率平衡下大有可为：

- **部署模式**：API 按量付费 vs 本地模型嵌入客户端直接运行
- **现状**：手机端跑本地模型能力有限（如 katago 28B 无法提交 App Store，需降级 18B 低参数运行）
- **趋势**：本地模型能力快速增长 + 手机算力持续提升
- **案例**：katago 18B 低参数仍可达准职业棋手棋力，单机已碾压 AlphaGo Lee 版本
- **建议**：关注开源模型部署和客户端集成，为未来 2-3 年的本地模型爆发做准备

## 自用开发趋势

越来越多的开发者不为赚钱做产品，而是市场上有但不满意（订阅费、弹广告、必须联网），于是用 AI 自己做一个免费、无广告、离线可用的版本。这种"搅局式开发"正在加剧 App Store 审核趋严。

## 未来趋势

- AI 从"辅助"到"主导"（90% AI + 10% 人工）
- 从"代码生成"到"架构设计"
- 从"单一模型"到"模型协作"
- 开发者角色从"编码者"转为"设计者+审查者"
