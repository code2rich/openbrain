# LLM-Wiki

Karpathy 提出的 AI 维护结构化知识库方法论，核心思想是「不要把 AI 当作更聪明的搜索引擎，而要把它当作永不疲倦的知识管理员」。

## 核心问题：传统 RAG 的缺陷

| 缺陷 | 说明 |
|------|------|
| 每次都在重新发现 | 同样的问题，AI 每次都要重新检索、重新理解 |
| 没有知识沉淀 | 今天的探索无法为明天的查询提供基础 |
| 缺乏全局视角 | AI 只能看到检索到的片段，无法建立系统性认知 |

**传统 RAG = 即时检索 + 临时综合 - 持久积累**

## 解决方案：从「检索」到「编译」

把 AI 从「检索助手」变成「知识管理员」——它不仅回答问题，还在每次交互中完善知识体系。

**Karpathy 原话**：「Wiki 是一个持久的、可复利的产物。交叉引用已经建立，矛盾已经标记，综合已经完成。Wiki 会随着你添加的每个来源、每个问题而变得更丰富。」

## 三层架构

```
原始资料库（Raw Sources）→ Wiki 知识库（The Wiki）→ Schema 配置（The Schema）
```

| 层级 | 说明 |
|------|------|
| 原始资料库 | 不可变真相层，AI 只读取从不修改 |
| Wiki 知识库 | AI 完全拥有的结构化知识层 |
| Schema 配置 | 系统运行规则层（CLAUDE.md） |

### Wiki 知识库结构

- **实体页面**：人物、公司、产品、概念的专门页面
- **主题综述**：某个领域的系统性总结
- **对比分析**：不同方案、观点的横向比较
- **index.md**：所有页面的目录 + 一句话摘要
- **log.md**：每次摄入、查询、维护的时间线记录

### 关键机制

当添加新资料时，AI 不是简单索引，而是：
1. 阅读并提取关键信息
2. 更新相关的实体页面
3. 修订主题综述
4. 标记与旧信息的矛盾
5. 建立新的交叉引用

这相当于「编译」——原始代码（文档）被转化为可执行程序（结构化 Wiki）。

## 四大工作流

### 1. 摄入（Ingest）

- 把文章放入 `raw/` 目录
- AI 阅读全文，与你讨论关键要点
- AI 在 Wiki 中创建摘要页面
- AI 更新 10-15 个相关的实体和主题页面
- AI 在 `log.md` 中记录这次摄入

### 2. 查询（Query）

- AI 查阅 `index.md` 找到相关页面
- AI 阅读已结构化的综合页面
- AI 生成答案（文档、对比表、图表）
- **关键**：好的答案被保存为新的 Wiki 页面

### 3. 归档（Filing）

将查询结果归档回 Wiki：「今天的探索 = 明天的基础设施」。

### 4. 维护（Lint）

定期健康检查：
- 不同页面之间有矛盾吗？
- 有被新资料推翻的旧观点吗？
- 有被频繁提及但缺少专门页面的概念吗？
- 有孤立页面（没有入链）吗？

## 成本对比

**传统 RAG**：每次查询成本 = 检索成本 + 综合成本（恒定）

**LLM-Wiki**：每次查询成本 = 初始编译成本 / 查询次数 + 小额维护成本（递减）

当查询次数超过 5-10 次，LLM-Wiki 就开始显现经济优势，且知识质量随时间提升。

## 为什么是 Markdown + Obsidian

| 工具 | 优势 |
|------|------|
| Markdown | AI 原生可读、完全可移植、人类友好 |
| Obsidian | 本地优先、双向链接、图谱视图、插件生态 |

**核心比喻**：Obsidian 是你的「IDE」，AI 是「程序员」，Wiki 是「代码库」。

## 适用场景

- 深度研究（AI、量子计算、投资策略）
- 阅读管理（为书籍章节建立伴读 Wiki）
- 竞品分析（持续追踪对比）
- 尽职调查（系统性信息整合）
- 团队内部知识库

## 生态工具：Graphify

详见 → [[Graphify]]

Graphify 直接解决了 Karpathy /raw 文件夹的痛点——每次查询都要重新读取全部文件、token 消耗巨大。核心是 tree-sitter AST + Claude 语义的双通道提取，52 个混合文件实测 71.5x token 压缩。

Graphify 侧重**图结构**（节点-边-社区），LLM-Wiki 侧重**文档结构**（实体-主题-对比），两者互补：Graphify 做初步图谱发现，LLM-Wiki 做深度知识提炼。

## 生态工具：Hermes Agent

详见 → [[HermesAgent]]

Hermes Agent 将 LLM Wiki 从「方法论」变成「开箱即用工具」，核心特色是微信原生集成（iLink Bot API 长轮询）和多 Wiki 管理。支持 `hermes wiki init` 一键创建、批量摄入、研究命令、多 Wiki 合并。Karpathy 推文 16 小时 1600 万次浏览后，Hermes 立刻实现了完整工作流。

## 实战案例：vault2wiki

[vault2wiki](https://github.com/悦然/vault2wiki)（作者：悦然）是 Karpathy LLM-Wiki 思路的一个完整落地实现，两天搭建完成。

### 核心设计原则

1. **原始材料与结构化知识分层**：raw 天然杂乱，不必一开始就强行整理成"完美条目"；分层避免每次输入都必须"整理到完美"的高成本
2. **可重复构建**：规则变了、模型变了、目录误删了，都能从 raw 全量重建。安全感来自"可重建"而非"从不出错"
3. **检索与写作变日常**：体系价值不在"存得多"而在"用得顺"，一键提问→答案写进 outbox→回流知识库，知识库才自增长

### 与本 Wiki 的架构对比

| 维度 | 本 Wiki | vault2wiki |
|------|---------|------------|
| 目录结构 | 00-raw / 99-wiki / 100-output（三层） | raw / wiki / outbox（三层） |
| 编译触发 | 手动触发（对话式） | 后台守护进程自动监听 |
| 质量控制 | AI 判断 + 人工审核 | 门控机制（元信息完整、正文非空等检查） |
| Glossary | 无 | 聚焦当前关注主题，非全量覆盖 |
| 安全网 | Git 版本管理 | 可重建能力（raw → wiki 全量重编译） |
| 模型容错 | 单模型 | 主端点失败自动切换备用模型 |

### 工程实现细节

**自动编译管线**：后台守护进程监听 raw 目录变化 → 触发编译流程 → 写入 wiki → 生成 verify/health 系统页 → state 记录增量状态避免重复处理。关键不是"自动"而是"可控地自动"：增量、可追踪、有失败信号。

**门控机制**：对结构化条目做质量检查（元信息完整、正文非空），不合格产物放入 outbox 隔离区。**导入阶段应放宽门控**——历史笔记常因标签不足、短文本被误拒，门控应保护系统而非成为输入阻力。

**标签策略**：不依赖模型理解分类意图，而是用可版本化的规则——文件名前缀硬映射主题标签 → 白名单与同义词映射清理，保证标签不失控扩散。

**Glossary 噪声处理**：笔记规模增长后出现噪声词条（长句、URL 被误当词条）和同义分裂（空格/大小写变体拆成多条）。解法：(1) 限制词条长度、过滤 URL (2) 合并写法变体 (3) 只从"当前关注主题范围"统计并提高生成阈值。Glossary 是雷达不是百科。

**灾难恢复**：作者实际遭遇过误删导致 wiki 全盘清空，依靠 raw → wiki 全量重编译恢复。可恢复性是系统底气。

### 关键实践洞察

1. **可重建 > 不出错**：知识库的安全感来自"raw 丢了能从 wiki 重建"的可重建能力，而非追求零错误
2. **门控保护系统但不阻碍输入**：导入阶段应放宽门控，避免标签不足等误拒卡住迁移
3. **Glossary 是雷达不是百科**：聚焦当前关注的主题，而非追求全量概念覆盖
4. **分类规则可版本化**：用文件名前缀映射主题标签 + 白名单同义词，不依赖模型理解
5. **工程细节决定长期体验**：模型降级、灾难恢复、增量编译这些"边界情况"才是系统长期运行的关键

### ex-brain：Compiled Truth 机制

详见 → [[ExBrain]]

ex-brain（作者：克军/OceanBase）在 Karpathy LLM-Wiki 基础上引入了"编译后的真相"概念，核心创新是按信息类型（状态/事实/事件）自动选择更新策略，底层数据库使用 SeekDB 混合检索。

### 与本 Wiki 的关键差异

| 维度 | 本 Wiki | ex-brain |
|------|---------|----------|
| 编译模式 | 手动（对话式 AI 处理） | `ebrain compile` 命令触发 |
| 知识更新策略 | AI 判断更新整页 | 按信息类型自动处理：状态→替换、事实→追加、事件→时间线 |
| 检索方式 | Obsidian 全文搜索 | seekdb 混合检索（向量 85% + 新鲜度 10% + 类型 5%） |
| 实体关联 | 手动 `[[双链]]` | LLM 自动识别关系并创建关联页面 |
| 时效性 | stale_score 人工审阅 | 信息衰减机制（设计方向，未实现） |

### 对本 Wiki 的启发

- **Compiled Truth 的分类处理**值得借鉴：本 Wiki 的 `knowledge_type`（permanent/periodic/ephemeral）可以更细粒度地指导更新策略
- **时间线自动抽取**：从 raw 文件中自动提取时间线事件，适合周期知识的时效追踪
- **seekdb 混合检索**：嵌入式部署、零运维，SQL 兼容，适合未来升级检索能力

## 实战案例：毛剑的「左移原则」实践

作者：毛剑（架构师的AI技术研究），开源工具链实现 LLM Wiki 全流程。

### 核心原则：左移（Shift-Left）

「入库就要高质量，问题越前置，成本越低」——不在后期修补质量，在摄入阶段就做去重、概念抽取、双链精选。

### 工具链

| 阶段 | 工具 | 职责 |
|------|------|------|
| 网页→Markdown | obsidian-skills (defuddle) | 网页净化 + 格式转换 |
| 图片本地化 | Local Image Plus | 远程图片下载到本地 |
| 双链增强 | obsidian-backlinks | 概念识别、归一化、精选 5-15 个双链 |
| Wiki 管理 | obsidian-wiki | 摄入/搜索/归档/健康检查四合一 |
| 自动化调度 | Openclaw Heartbeat | 定时执行健康检查 |

### obsidian-wiki 四大功能

1. **新增（入库）**：去重检测 → 摘要+概念提取 → 概念联网搜索 → 判断新建/完善 → 更新索引
2. **搜索（检索+问答）**：检索 Wiki（概念/摘要/问答/Index）→ 必要时联网补充 → 多轮对话 → 推理回答
3. **归档（知识沉淀）**：高质量 QA 自动归档到问答库，问答库本身也成为检索源
4. **健康检查**：概念新鲜度、References 完整性、双链一致性、索引一致性、同义词合并

### 与本 Wiki 的关键差异

| 维度 | 本 Wiki | 毛剑方案 |
|------|---------|----------|
| 目录策略 | 00-raw / 99-wiki / 100-output 三层分离 | 平铺目录 + 双链 + Tags |
| 双链策略 | AI 判断关联，数量不限 | 每篇精选 5-15 个，宁缺毋滥 |
| 问答沉淀 | 好的查询结果保存为 Wiki 页面 | QA 归档到独立问答库 |
| 健康检查 | AI 对话式维护 | Openclaw Heartbeat 定时自动执行 |
| 知识检索 | Obsidian 全文搜索 | Wiki 检索 + 联网搜索混合 |

### 对本 Wiki 的启发

- **精选双链**：「好的双链不在多，在于精准」——本 Wiki 可考虑对单页双链数量设上限
- **问答库独立存储**：将 QA 与概念文章分离，问答库作为独立检索源，避免知识页面膨胀
- **自动化健康检查**：Openclaw Heartbeat 的定时模式值得参考，减少人工触发维护的负担

## 实战案例：陆离同学的 9830 条记忆教训

详见 → [[ai-memory-governance]]

作者从 4 年 Obsidian 日记中提取 9830 条记忆，先接入 MemOS（Qdrant + Neo4j + MCP），最终推倒重来，改为纯文本治理方案（SOUL.md + USER.md + self-improving/）。

**核心验证**：记忆系统的关键不是存储量，而是治理结构。复杂基础设施（向量库+图数据库+MCP Server）的维护成本（25-35 分钟/天）远超纯文本方案（接近 0），而后者在知识治理效果上反而更好。

这是从"反面教材"验证 LLM-Wiki 方法论的重要案例：好答案不应消失在对话里，高价值内容必须进入可持续更新的结构化知识库。
