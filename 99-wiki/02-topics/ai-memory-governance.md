# AI 记忆治理

## 核心洞察

> 记忆不是越多越好，是"有效的治理"才是最好。

AI 长期助手的核心挑战不在于存储量，而在于**治理结构**：高价值信息有没有被持续整理、持续更新、持续写回。如果没有，记忆再多也只是堆在那里——每次问问题，系统还是临时抓材料拼凑。

### 好答案不应该消失在对话里

- 高价值内容应该进入知识库
- 系统的重点不是一遍遍检索，而是持续维护一个能被调用、能被更新、能被交叉引用的长期结构
- 真正重要的经验、判断、方法和长期有效的结论，不应散在各种聊天和日志里消失掉

## 踩坑实录：9830 条记忆的教训

### 背景

作者（陆离同学）花了三个月，从 4 年 Obsidian 日记中提取了 9830 条记忆，喂进 OpenClaw + MemOS 的记忆系统，最终推倒重来。

### 第一阶段：MemOS 方案（复杂架构）

| 组件 | 用途 |
|------|------|
| Qdrant | 向量数据库（语义搜索） |
| Neo4j | 图数据库（关系图谱） |
| MCP Server | 记忆查询接口 |
| Worker | 同步脚本 |
| 数据库 × 2 | 双存储引擎 |

**踩坑清单**：

| 问题 | 描述 |
|------|------|
| 日记更新断层 | 谁来判断哪些是偏好、哪些是知识沉淀、哪些是流水账？没有这套机制，记忆库会过期 |
| 时间戳冲突 | 批量导入时所有记忆变成同一天，失去时间维度 |
| MCP 不常驻 | 每次新会话要手动检查服务状态，进程经常消失 |
| 双库不同步 | Qdrant 有 11,000+ 向量，Neo4j 一条没有，语义搜索和图谱查询走两套互不相干的数据 |
| 维护成本高 | 每天 25-35 分钟 |

### 第二阶段：推倒重建（纯文本方案）

受 Claude Code 源码泄漏启发（分析了规则组织、行为约束、记忆与工作流的统一框架），得出结论：**"核心问题不是配置问题，是架构错了。"**

## 推荐架构：三层治理

```
SOUL.md          → 定义 AI 人格和语气
USER.md          → 定义用户画像、偏好、协作方式
self-improving/  → AI 纠正记录、成功经验、错误沉淀
```

### 核心组件

| 文件 | 职责 | 特点 |
|------|------|------|
| SOUL.md | AI 人格定义 | 定义 AI 的性格、语气、行为风格 |
| USER.md | 用户画像 | 偏好、协作方式、习惯 |
| self-improving/ | 自我改进记录 | 错误沉淀 + 成功经验 + 行为纠正 |

### 关键设计原则

1. **纯文本**：没有向量库、没有 Docker、没有 MCP Server，Claude Code 和 OpenClaw 直接读
2. **治理优于存储**：关注高价值信息的持续整理、更新、写回
3. **渐进式启动**：先填好 SOUL.md 和 USER.md，建 self-improving/ 目录，其余按需添加

## 方案对比

| 维度 | MemOS 方案 | 纯文本治理方案 |
|------|-----------|--------------|
| 维护成本 | 25-35 分钟/天 | 接近 0 |
| 技术依赖 | Qdrant + Neo4j + MCP + Worker | 无 |
| 数据同步 | 双库需手动同步 | 单一文件系统 |
| 可维护性 | 高运维复杂度 | 纯文本，人类可读 |
| 适用场景 | 需要语义搜索+图谱的复杂场景 | 个人助手、知识管理 |

## 开源项目

[openclaw-memory-system](https://github.com/Elo-Mario/openclaw-memory-system) — 完整的记忆+进化+做梦系统，纯文本架构，零依赖。

- 都采用**纯文本 + Markdown** 作为载体
